# Delegate worker skill discipline to shared engineering skills

The bundled PullOps worker skills stop restating engineering procedure and delegate the _how_ to the shared engineering skills: `pullops-issue-implement` works the issue as `/implement` describes (with `/tdd`, `/coding-standards`, `/diagnosing-bugs` where they apply), `pullops-pr-review` reviews along the two `/code-review` axes including its Fowler smell baseline, and `pullops-pr-address-review` names the same disciplines in one line instead of a per-discipline checklist. What stays in each worker skill is exactly what only PullOps can say: the operation boundary, the liveness and command-gate rules (`pullops step`, heartbeats), the scope rules (Adjacent Work, `followUps`), and the JSON Operation Output contract that `pullopsSkillContracts.test.js` pins against the generated prompts.

The July 2026 audit flagged ~930 lines of prescriptive installed skills dual-maintained against operation prompts. Step-by-step procedure ("inventory the context before judging", "run focused verification") is knowledge-encoding a competent runner already carries — the Bitter Lesson cut from ADR-0066 — and every restated procedure line was a second copy that could drift from the shared skill teaching the same thing. Contracts age well; procedures age badly.

Delegation degrades gracefully: each worker skill still states the essence of the delegated discipline in a sentence (the two review axes, the tdd-at-seams loop), so a Target Repository that has not installed the optional engineering skills still gets a correct, just less detailed, briefing. The shared skills sharpen the work where they exist; the PullOps contract holds either way.
