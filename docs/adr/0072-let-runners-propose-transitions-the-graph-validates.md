# Let runners propose transitions the graph validates

The PullOps-Managed PR Transition Graph now works as a validator, not only a driver. An Operation Output may carry a **Proposed Next Operation**; `resolveNextManagedPrOperation` in `src/managed-pr/transitionPolicy.js` applies the proposal when it is an allowed continuation of the reported outcome and silently falls back to the graph's default edge otherwise. Allowed continuations are the default edge plus explicitly declared detours (`PROPOSABLE_MANAGED_PR_DETOURS`); every detour still funnels back through `pr-review`, so a proposal can widen the route but never skip verification or the human merge boundary, and a proposal can never revive a terminal outcome such as `blocked`.

This executes the preparation recorded in ADR-0069 ("expose an isTransitionAllowed-style check when the model starts proposing next operations") and the ADR-0066 split: which operations _may_ follow which outcome stays verified harness structure, while which allowed continuation is _best_ becomes runner judgment. The alternative — encoding each routing nuance as a new outcome kind, the way `pr-finalize` grew `route-to-review` and `route-to-ci-fix` — makes the outcome vocabulary carry routing decisions and needs a graph edit for every nuance a stronger model could simply judge.

The first proposal surface is `pr-review`: a changes-requested review may set `nextOperation` to `pr-fix-ci` when the real blocker is failing checks, instead of bouncing a `pr-address-review` pass off red CI. Output validation accepts only managed-PR workflow operation names and only alongside `changes_requested` — schema checks taxonomy, the graph checks allowance. Detours are added to `PROPOSABLE_MANAGED_PR_DETOURS` deliberately, one reviewed edge at a time, and graph tests assert every allowed continuation short of an approving review stays away from `pr-finalize`.
