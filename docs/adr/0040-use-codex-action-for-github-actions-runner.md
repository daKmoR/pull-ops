# Use Codex Action as an explicit GitHub Actions runner adapter

PullOps keeps local operation runs on the configured `codex-cli` runner adapter by default, while GitHub Actions workflows select `codex-action` explicitly for Codex-backed operations. A GitHub Action is a workflow step rather than a process-level runner, so implement and review operations split into prepare/action/finalize phases only for that adapter, with no deprecated runner-coupled phase names in the unreleased CLI. The `codex-action` adapter is invalid without an explicit prepare or finalize phase, and package scripts should expose only human/local operation entrypoints rather than lifecycle helper scripts. PullOps still owns issue and PR orchestration, prompt construction, structured output validation, commits, labels, and comments, while the Codex Action owns installing, authenticating, and executing Codex. Prepare exposes whether the runner step should execute as the generic `run_runner` workflow output, and workflow scratch-space remains an `OUTPUT_DIR` environment variable rather than operation input. Codex Action jobs keep the built-in workflow token read-only; PullOps prepare and finalize receive `PULLOPS_GITHUB_TOKEN` explicitly, and finalize sets authenticated git origin only immediately before post-run mutations. Operation workflows allow `github-actions[bot]` to run the Codex Action because the dispatcher creates them through `workflow_dispatch`; the dispatcher preserves Codex Action's write-access intent by verifying the original label actor before dispatching. Finalize restores the repository's configured Node version after the Codex Action because the action installs its own Node runtime. Finalize still runs when prepare skips the runner, but `--runner-ran false` is a no-op lifecycle acknowledgement rather than a second validation pass. When the Codex Action ran and failed, finalize records the PullOps failure state before exiting nonzero.
