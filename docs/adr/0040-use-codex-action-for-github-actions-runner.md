# Use External Runner Adapter for GitHub Actions runner handoff

PullOps keeps local operation runs on the configured `codex-cli` runner adapter by default, while GitHub Actions workflows select the host-neutral `external` runner adapter for Codex-backed operations. A GitHub Action is a workflow step rather than a process-level runner, so implement and review operations split into prepare/runner/complete phases only for that adapter, with no deprecated runner-coupled phase names in the unreleased CLI. The `external` adapter is invalid without an explicit prepare or complete phase, and package scripts should expose only human/local operation entrypoints rather than lifecycle helper scripts. PullOps still owns issue and PR orchestration, prompt construction, structured output validation, commits, labels, and comments, while the external runner step owns installing, authenticating, and executing Codex. Prepare exposes whether the runner step should execute as the generic `run_runner` workflow output, writes neutral `runner_prompt.md` and `runner_output.json` artifact paths, and workflow scratch-space remains an `OUTPUT_DIR` environment variable rather than operation input. A manager-owned `pullops runner-result` command writes `runner_result.json` before complete so PullOps can distinguish success, failed, cancelled, and skipped runner states before reading operation output. External runner jobs keep the built-in workflow token read-only; PullOps prepare and complete receive `PULLOPS_GITHUB_TOKEN` explicitly, and PullOps CLI git fetch and push operations configure authenticated `origin` from the PullOps token when GitHub Actions repository env is present. Operation workflows check that `OPENAI_API_KEY` is configured before invoking the Codex Action so setup failures do not surface as action-internal proxy metadata errors. Operation workflows allow `github-actions[bot]` to run the Codex Action because the dispatcher creates them through `workflow_dispatch`; the dispatcher preserves Codex Action's write-access intent by verifying the original label actor before dispatching. Codex Action `codex-args` set `approval_policy` through `--config` instead of `--ask-for-approval` because the action appends extra arguments after `codex exec`, where current Codex CLI versions accept config overrides but reject that global flag. Complete restores the repository's configured Node version after the Codex Action because the action installs its own Node runtime. Complete still runs when prepare skips the runner, and the skipped `runner_result.json` is a no-op lifecycle acknowledgement rather than a second validation pass. When the external runner reports failed or cancelled, complete records the PullOps failure state before exiting nonzero.
