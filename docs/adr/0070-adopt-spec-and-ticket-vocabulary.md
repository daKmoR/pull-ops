# Adopt spec and ticket vocabulary from the skills v1.1 flow

PullOps names its two work shapes with the vocabulary of the mattpocock/skills v1.1 authoring flow: a **Spec Issue** (formerly PRD Issue) is the published destination for a larger change, and **Tickets** (formerly Child Issues) are the tracer-bullet vertical slices cut from it. The upstream flow is grilling → spec → tickets → implement → code review; PullOps operations pick up exactly where `/to-spec` and `/to-tickets` publish, so the tracker terms, operation labels, CLI references, progress events, and glossary all use the same two words the authoring skills use. The renames follow upstream's v1.1 reasoning: what `/to-prd` produced was never a product requirements document — it is a specification, technical or not — and "issues under a spec" are more honestly the general concept "tickets" than GitHub's tracker-specific noun.

This is a rename with no compatibility layer, taken while nothing is released. The `prd` Operation Label target kind becomes `spec` (`pullops:spec:prepare`, `pullops:spec:auto-advance`, `pullops:spec:auto-complete`), operation names and workflow files follow (`spec-prepare`, `spec-auto-advance`, `spec-auto-complete`, `pr-close-ticket`), issue publication commands become `issues publish-spec` and `issues publish-tickets`, ticket-scoped progress events become `ticket.started`/`ticket.progress`/`ticket.completed`/`ticket.blocked` in the `ticket-coordination` phase, the commit footer for umbrella history becomes `Spec: #<n>`, and the sink env var for the worked ticket becomes `PULLOPS_TICKET_NUMBER`.

The rename deliberately stops at the run-supervision boundary. "Child" survives where it names a nested _run_, not a work item: `child.heartbeat` events, Child Heartbeat Event, `childRun` links and state paths, `PULLOPS_CHILD_RUN_ID`, and the Parent Event Sink's child routes all describe the parent/child relationship between a supervising command and the worker it spawned (which may be working a ticket or an umbrella PR). Collapsing that into "ticket" would misname umbrella PR runs and blur ADR-0064's separation between liveness signals and semantic progress. So: tickets are what the work is, child runs are how it executes.
