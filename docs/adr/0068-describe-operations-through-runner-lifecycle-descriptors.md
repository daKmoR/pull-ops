# Describe operations through Runner Lifecycle descriptors

Every runner-backed Operation Module hands the Runner Lifecycle one Operation Descriptor — its runner-step factories, finalize ordering, and local dry-run flow as data — and the Runner Lifecycle owns the shared execution flow: the local dry-run guard for pull request targets, run/prepare/complete phase dispatch, and finalize ordering. Before this decision each Operation Module re-implemented that flow in copy-pasted entry, prepare, and finalize exports, so a lifecycle fix meant editing six modules and a new operation started from scaffolding instead of a descriptor.

Operation Modules keep everything ADR-0027 gives them: orchestration code, prompts, extraction instructions, Operation Output schemas, and genuinely bespoke flows. A descriptor may override individual phases — `pr-resolve-conflicts` keeps its multi-pass Conflict Resolution loop and bespoke external phases, and `issue-implement` keeps its issue-target entry dispatch — but the override is declared descriptor data, not parallel scaffolding.

The named entry exports (`runPrReview`, `runPrReviewExternalRunnerPrepare`, …) remain as one-line phase dispatches through the descriptor, so the Operation Catalog's handler imports and direct callers such as tests and PRD automation keep working unchanged. This preserves ADR-0049's catalog shape: the catalog still imports operation-owned handlers and owns no execution flow. An Operation Descriptor is a Runner Lifecycle interface, not the catalog-facing identity descriptor that ADR-0049 rejected.
