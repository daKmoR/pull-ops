# Separate execution backend from publication mode

PullOps local execution should use an explicit Execution Backend, Run Goal, and Publication Mode rather than overloading Runner Adapter flags with user intent. `pullops run` is the Human-Facing Command for running a short Operation Label Reference such as `issue:implement` against a numeric target, with the target kind inferred from the reference, while GitHub Actions lifecycle plumbing moves to Workflow-Facing Commands outside that surface. Runner Adapter remains lower-level than Execution Backend: it describes how AI is invoked inside a backend, not where the operation is driven from. Human-facing terminal runs default to the local backend, the Operation Run Goal, and dry-run publication; the local command itself is the request and does not require the Operation Label to already be present. The GitHub Actions backend applies the requested Operation Label and lets the normal label-dispatch path run. Publication Mode and Run Goal apply only to the local backend; GitHub Actions runs reject publication and `--until` flags because the label-dispatch workflow owns that behavior. The initial Run Goals are Operation and Finalized, selected in the CLI with `--until`: Operation runs exactly the requested operation, while Finalized follows local PullOps follow-up operations until the pull request is finalized or blocked. The local backend runs in the active Target Repository checkout, fetches relevant remote refs before starting, requires a clean worktree, uses normal PullOps branches for both dry-run and pull request publication, invokes AI through the configured Runner Command, and uses operation-configured model selection. Local execution writes a visible gitignored Local Run Record under `.pullops/runs/` for every local run, named with a timestamp, normalized Operation Label Reference, and target number, so prompts, outputs, logs, patches, review comments, and intermediate results can be inspected, passed between local follow-up operations, and replayed as relevant pull request evidence after PR creation without mutating GitHub during dry-run; users clean these run records manually until PullOps has a dedicated cleanup command. Publication runs create a fresh Local Run Record by default rather than reusing earlier records, but may publish the current clean worktree without rerunning AI when it is already on the expected PullOps branch with local commits for the target. Dry-run publication never mutates GitHub state; operation-only and finalized dry-runs may create local commits and leave the PullOps branch checked out so the maintainer can review AI-written commit messages and inspect the branch without pushing it. Local pull request publication performs real GitHub state transitions using the locally authenticated user, creates or updates normal PullOps-Managed PR state, records Operation Audit Comments as evidence without using trigger labels for continuation, and single-target runs leave the result branch checked out. For the Finalized Run Goal, PullOps opens or updates the pull request only after the local follow-up pipeline completes and marks it ready for review, while operation-only publication keeps the existing draft PR behavior; audit evidence should not force early PR creation and may be reduced if there is no simple place to record intermediate local operation evidence. Blocked local runs leave the checkout on the blocked branch state for inspection instead of cleaning up or resetting. PRD auto-advance keeps the same human review gates locally as in GitHub Actions: dry-run stops at one runnable Child Issue, while pull request publication can implement currently unblocked Child Issues sequentially, return to a clean PRD base between them, and leave the Umbrella Branch checked out.
