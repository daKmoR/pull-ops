# Separate PR state, links, and operation audit

PullOps pull request bodies should stay useful to human reviewers while still carrying the current resumable workflow state. The body keeps a compact visible PR State Marker (`Managed: yes` and `Status: ...`), an always-visible PullOps Link Summary that emphasizes related pull requests over issue lists, and a collapsed PullOps workflow state block for cycle counts, last operation, and merge-preparation hashes; runner metadata, model information, reasoning effort, and known context usage move to append-only Operation Audit Comments instead of the PR body.

Issue bodies remain specifications and should not gain generated PullOps status sections. PullOps uses the PR body as the source of truth for pull request workflow status, natural GitHub state for completed issues and merged pull requests, and a single `pullops:human-required` Status Label only when maintainer attention is needed. This supersedes the parts of ADR-0021 and ADR-0039 that put trigger context and ordinary progress/completion status labels in the primary PullOps state model.

PullOps should only treat a pull request as managed when the visible `## PullOps` block contains `Managed: yes` and `Status: ...`, and it should parse diagnostic workflow markers only inside a `<details><summary>PullOps workflow state</summary>` block. Ticket PR link summaries show `Source Issue` and `Umbrella PR`, using `Umbrella PR: pending` until the umbrella pull request exists; Umbrella PR link summaries show the Spec Issue, `Closes: #<number>`, and Ticket PRs, using `Ticket PRs: none yet` before ticket pull requests exist. The `pullops:human-required` label is removed when PullOps accepts a retry operation and reapplied if the retry again needs maintainer attention.
