# Keep policy in prompts and config, verification in code

PullOps encodes judgment as prompt and configuration data and reserves code for verification and trust boundaries, so runner-model improvements flow into PullOps behavior without code changes. Policy — how to classify a failure, how to plan a repair, which approach to take — belongs in Operation Module prompts, installed PullOps Skills, and PullOps Config, where a stronger model can immediately do better and a maintainer can tune without touching code. Code owns what must not drift with model quality: Operation Output validation, deterministic verification of runner results (such as diff inspection before committing a CI repair), resource and cycle guards, and human approval boundaries. When a rule can be checked deterministically after the runner acts, PullOps verifies the result instead of prescribing the procedure in prompt prose; when a rule is a value or trust boundary, the prompt states it and code still enforces it. The first application is `pr-fix-ci`: the runner owns Check Failure Classification with the keyword classifier demoted to a recorded non-binding prior, and deterministic working-tree verification refuses repairs that delete or skip tests, remove assertions, or alter check and workflow configuration, regardless of self-reported safety flags.
