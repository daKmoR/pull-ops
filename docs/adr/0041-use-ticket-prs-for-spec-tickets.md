# Use ticket PRs for Spec tickets

Spec Issues use an Umbrella Branch named `pullops/spec-<spec-number>` and an Umbrella PR that targets the default branch, while each Ticket uses `pullops/spec-<spec-number>-issue-<issue-number>` and opens a Ticket PR targeting the Umbrella Branch. The ticket branch shape deliberately avoids nesting under the umbrella branch name because git cannot store both `refs/heads/pullops/spec-<spec-number>` and `refs/heads/pullops/spec-<spec-number>/...` at the same time. Ticket PR bodies avoid GitHub closing keywords because [GitHub ignores those keywords for PRs targeting non-default branches](https://docs.github.com/en/issues/tracking-your-work-with-issues/using-issues/linking-a-pull-request-to-an-issue); they use non-closing references such as `Refs #<issue>` and `Part of #<spec>`. PullOps closes the Ticket from a deterministic `pull_request.closed` workflow only after a merged same-repository Ticket PR targets the matching Umbrella Branch, and dependency checks treat `Blocked by: #<issue>` as satisfied only when the blocking issue is closed.

The final Spec Issue remains open until the Umbrella PR merges into the default branch. Once the last native Ticket is closed, PullOps requests automated review on the Umbrella PR by applying `pullops:pr:review`; approved review then follows the normal PR Finalize path. ADR-0044 introduces `spec-auto-complete` as the hands-off automation path that drives Tickets through implementation, review, finalization, and Umbrella Branch integration, then reviews and finalizes the Umbrella PR while leaving its default-branch merge human-controlled.
