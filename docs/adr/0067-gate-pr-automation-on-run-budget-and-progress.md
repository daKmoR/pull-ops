# Gate PR automation on Run Budget and progress

PullOps-Managed PR automation continues while the target's Run Budget has capacity and each cycle shows verifiable progress, instead of stopping at fixed cycle counts. The Run Budget is configured per Target Repository in PullOps Config (`runBudget.maxUsedTokens` and `runBudget.maxDurationMs`) and is charged from runner-reported Context Usage and measured per-operation wall-clock time; accumulated usage is recorded as `Run budget used tokens` and `Run budget used ms` markers in the PullOps Workflow State block so gating survives resumption. Progress verification is deterministic: a changes-requested review records the rejected tree as a `Last cycle tree` marker, a later review on an unchanged tree blocks as no-progress, and operations that change the tree (address-review, fix-ci) clear the marker. Human Feedback Response Cycles skip the no-progress check because trusted human feedback is new input, and Escalation Review Cycles and Human Feedback Response Cycles consult the budget before starting. Cycle counters remain recorded as telemetry and act only as generous backstop guards (ten Review Cycles, six CI Fix Cycles by default), so a budget accounting bug can never cause an unbounded loop. Blocked outcomes report structured PullOps Run Blocker kinds — `budget-exhausted`, `no-progress`, or `safety-refusal` — so a maintainer immediately knows what action is theirs.
