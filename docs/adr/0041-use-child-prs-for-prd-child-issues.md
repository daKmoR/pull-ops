# Use child PRs for PRD child issues

PRD Issues use an Umbrella Branch named `pullops/prd-<prd-number>` and an Umbrella PR that targets the default branch, while each Child Issue uses `pullops/prd-<prd-number>-issue-<issue-number>` and opens a Child Issue PR targeting the Umbrella Branch. The child branch shape deliberately avoids nesting under the umbrella branch name because git cannot store both `refs/heads/pullops/prd-<prd-number>` and `refs/heads/pullops/prd-<prd-number>/...` at the same time. Child Issue PR bodies avoid GitHub closing keywords because [GitHub ignores those keywords for PRs targeting non-default branches](https://docs.github.com/en/issues/tracking-your-work-with-issues/using-issues/linking-a-pull-request-to-an-issue); they use non-closing references such as `Refs #<issue>` and `Part of #<prd>`. PullOps closes the Child Issue from a deterministic `pull_request.closed` workflow only after a merged same-repository Child Issue PR targets the matching Umbrella Branch, and dependency checks treat `Blocked by: #<issue>` as satisfied only when the blocking issue is closed.

The final PRD Issue remains open until the Umbrella PR merges into the default branch. Once the last native Child Issue is closed, PullOps requests automated review on the Umbrella PR by applying `pullops:pr:review`; approved review then follows the normal PR Finalize path. ADR-0044 introduces `prd-auto-complete` as the hands-off automation path that drives Child Issues through implementation, review, finalization, and Umbrella Branch integration, then reviews and finalizes the Umbrella PR while leaving its default-branch merge human-controlled.
